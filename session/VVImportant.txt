1.Test Harness হলো এমন একটা framework / পরিবেশ, যেখানে সফটওয়্যার বা কোনো নির্দিষ্ট  module/component-কে টেস্ট চালানো, ইনপুট দেওয়া, আউটপুট যাচাই করা এবং ফলাফল রিপোর্ট করা হয়.
একটা Test Harness সাধারণত এসব দিয়ে তৈরি হয়:Test Script / Test Case,Test Data,Test Driver,Stub / Mock,Test Execution Tool,Test Result / Report.

2.Software Testing-এ “Work Products” মানে হলো—টেস্টিং প্রক্রিয়ার সময় আমরা যে ডকুমেন্ট, রিপোর্ট, আর্টিফ্যাক্ট বা আউটপুট তৈরি করি, সেগুলোই Work Products।
সহজ ভাষায়:
👉 Testing করতে গিয়ে যেসব জিনিস লিখে/তৈরি করে রেখে দিই—সেগুলোই Work Products.

3. Testware বলতে সফটওয়্যার টেস্টিং-এ ব্যবহৃত সব ধরনের ডকুমেন্ট, টেস্ট আর্টিফ্যাক্ট এবং সাপোর্টিং উপকরণকে বোঝায়, যেগুলো টেস্টিং কার্যক্রম পরিকল্পনা, ডিজাইন, এক্সিকিউশন এবং রিপোর্টিং-এ কাজে লাগে.
সহজভাবে বললে
👉 Test Ware = টেস্ট করার জন্য যা কিছু বানানো বা ব্যবহার করা হয়.

4. 👉 Test Object হলো সেই নির্দিষ্ট জিনিস/এলিমেন্ট/ফিচার যেটাকে আমরা টেস্ট করি.

সহজভাবে বললে—
🔹 যেটার উপর টেস্ট চালানো হয়, সেটাই Test Object.

5. Static Testing Review process- Planning,Review initiation, Individual review, Communication and analysis, Fixing and reporting.
--Planning (পরিকল্পনা)
→ Review-এর scope (কী কী রিভিউ হবে), schedule (সময়), এবং participants নির্বাচন করা হয়,Reviewer, moderator, author, scribe ইত্যাদি ভূমিকা নির্ধারণ করা হয়
➡️ এই ধাপে পুরো রিভিউ কীভাবে চলবে তার প্ল্যান করা হয়।

--Kick-off (শুরু সভা)
Review-এর objectives ব্যাখ্যা করা হয় এবং documents বিতরণ করা হয়।
➡️ সবার কাছে পরিষ্কার করা হয়—কেন রিভিউ করা হচ্ছে ও কী রিভিউ করা হবে।

--Individual Preparation (ব্যক্তিগত প্রস্তুতি)
→ প্রত্যেকে আলাদা আলাদা ভাবে ডকুমেন্ট পড়ে defects খোঁজে।
➡️ এখানে মিটিং ছাড়া নিজের মতো করে ভুল ধরার কাজ হয়।

--Review Meeting (রিভিউ মিটিং)
→ সবাই মিলে defects আলোচনা করা হয় এবং log করা হয়।
➡️ এটাই মূল রিভিউ মিটিং।

--Rework (সংশোধন কাজ)
→ ধরা পড়া defects ঠিক করা হয়।
➡️ Author বা ডেভেলপার ভুলগুলো ফিক্স করে।

--Follow-up (অনুসরণ)
→ Fix করা হয়েছে কিনা verify করা হয় এবং metrics সংগ্রহ করা হয়।
➡️ রিভিউ সফল হয়েছে কিনা তা নিশ্চিত করা হয়

6. Test Basis (টেস্ট বেসিস) হলো সেই ডকুমেন্ট, তথ্য, বা সোর্স যেগুলোর উপর ভিত্তি করে আমরা টেস্ট কেস ডিজাইন করি।

7. Failure rate (সঠিক উত্তর) এটি হলো নির্দিষ্ট সময়, ব্যবহার বা মাপের একক অনুযায়ী কতবার ব্যর্থতা ঘটছে তার অনুপাত।
   উদাহরণ: প্রতি ঘন্টায় গড়ে কতবার সিস্টেম ব্যর্থ হচ্ছে

8. Fault Attack হলো একটি টেস্টিং টেকনিক যেখানে ইচ্ছাকৃতভাবে সিস্টেমে ত্রুটি বা ব্যর্থতা ঘটানোর চেষ্টা করা হয়, যাতে দেখা যায় সিস্টেম কীভাবে প্রতিক্রিয়া জানায়.
   👉 "They can evaluate the reliability of a test object by attempting to force specific failures to occur."
   অর্থাৎ, fault attack ব্যবহার করে আমরা যাচাই করতে পারি একটি সিস্টেম কতটা নির্ভরযোগ্য, কারণ আমরা ইচ্ছাকৃতভাবে ব্যর্থতা ঘটিয়ে দেখি সেটি সামলাতে পারে কিনা.

9.Memory leak হলো runtime সমস্যা। এটি সাধারণত Performance Testing বা Reliability Testing-এর অংশ but also Component Testing-এ memory leak পরীক্ষা করা হয়.

10.Basic Path Testing:এটি একটি white-box technique, যেখানে কোডের সব path (execution path) এবং condition কভার করার চেষ্টা করা হয় but Branch Testing → এটি কোডের সব branch (if/else) কভার করে, কিন্তু সব path এবং condition কভার করে না. 

11.Test incident report identifier →(Unique identifier) প্রতিটি incident report-এর জন্য একটি আলাদা ID থাকতে হবে।
Summary →(References to other relevant documents) Summary অংশে incident-এর সাথে সম্পর্কিত অন্যান্য ডকুমেন্টের রেফারেন্স থাকে।
Incident description →(Anomalies, Procedure step, Environment)এখানে সমস্যার বর্ণনা, কোন ধাপে সমস্যা হয়েছে, এবং কোন environment-এ হয়েছে তা লেখা হয়।
Impact →(Impact on test plans) Incident-এর কারণে test plan বা execution-এর উপর কী প্রভাব পড়ছে তা এখানে উল্লেখ করা হয়

12. Static Testing Review Types: 
Informal Review-> Undocumented, এটি সবচেয়ে অনানুষ্ঠানিক, কোনো ডকুমেন্টেশন থাকে না.
Walkthrough->Led by author, লেখক নিজেই কোড/ডকুমেন্ট উপস্থাপন করেন এবং peer group feedback দেয়, One kind of Informal, Preparation optional.
Peer Review/Technical Review-> No management participation, সহকর্মীরা একে অপরের কাজ রিভিউ করে।
Inspection-> Led by a trained moderator or leader, Uses entry and exit criteria,এটি সবচেয়ে formal review process, Formal defect detection by independent reviewers,  Managers can perform inspections on management documents, Inspection compares documents with predecessor (source) documents.

13. Test oracle সবসময় external reference হয়, যেমন user manual, পুরনো সিস্টেম, বা domain expert-এর জ্ঞান but not for the code.

14.Test Case Specification → টেস্ট কেসের বিস্তারিত,Test Procedure Specification → টেস্ট চালানোর ধাপ,Test Plan → টেস্টিং-এর পরিকল্পনা,Test Design Specification → টেস্ট কন্ডিশন ও high-level কে. Majority of system errors occur in the Requirements phase.

15. Testing Stop when scheduled time for testing expires, প্রকল্পের পরিকল্পনা অনুযায়ী নির্দিষ্ট সময়সীমা থাকে,সেই সময় শেষ হলে টেস্টিং থামাতে হয়, এমনকি সব বাগ ধরা না পড়লেও, টেস্টিং সবসময় সীমাহীনভাবে চলতে পারে না.

16. Test Summary Report-এ নিচের সেকশনগুলো থাকে:Variances → পরিকল্পনা অনুযায়ী কী কী ভিন্নতা ঘটেছে তা উল্লেখ করা হয়, Comprehensive assessment → টেস্টিং-এর সামগ্রিক মূল্যায়ন দেওয়া হয়, Evaluation → টেস্টিং-এর ফলাফল এবং সিস্টেমের গুণগত মান মূল্যায়ন করা হয়, Summary of activities → টেস্টিং চলাকালীন কী কী কার্যক্রম হয়েছে তার সারসংক্ষেপ দেওয়া হয়।

17. Incident report তৈরি বা discrepancies রিপোর্ট করা সবসময় Test Execution phase-এর অংশ, Software Release-এর পর user community-তে change দেওয়ার পর acceptance testing করা হয় না. Functional Testing = Validation technique কারণ এটি requirements অনুযায়ী সিস্টেমের সঠিকতা যাচাই করে and Are we building the right product?” → অর্থাৎ সিস্টেমটি ব্যবহারকারীর চাহিদা পূরণ করছে কিনা and Verification এটি design, documentation, এবং coding standard যাচাই করে Are we building the product right?

18. Stub = Skeletal implementation of a software component for testing। এটি একটি dummy module যা আসল module-এর behaviour simulate করে। Stub সাধারণত ব্যবহার হয় → Top-Down Integration Testing.
Driver একটি dummy program এটি module/function কে call করে এবং test data pass করে and Driver ব্যবহার হয় → Bottom-Up Integration Testing.  
Preventative tests are designed early; reactive tests are designed after the software has been produced. PDCA = Continuous Improvement Cycle → Plan → Do → Check → Act

19. Document review process-এ নেতৃত্ব দেয় Moderator, Moderator- planning, meeting পরিচালনা এবং follow-up-এর দায়িত্বে থাকে and প্রতিটি program function operational কিনা তা যাচাই করার জন্য টেস্ট কেস তৈরি করা হয় Grey-box Testing-এ. Process improvement হলো review result ব্যবহার করে future process উন্নত করা, যা review activity এর সরাসরি cost না।

20. Test exit criteria হলো সেই শর্তগুলো যা পূরণ হলে আমরা বলতে পারি টেস্টিং শেষ হয়েছে। Thoroughness measures → যেমন coverage criteria (statement, branch, requirement coverage) ✔️ Reliability measures → সিস্টেম কতটা স্থিতিশীল তা যাচাই করা ✔️ Test cost এবং schedule → বাজেট ও সময়সীমা পূরণ হয়েছে কিনা ✔️ State of defect correction → critical defect ঠিক করা হয়েছে কিনা ✔️ Residual risks → বাকি থাকা ঝুঁকি গ্রহণযোগ্য কিনা.

21. The broad and primary purpose of reviewing a deliverable is defect detection in any work product (requirements, design, code, test cases, etc.).While spelling mistakes (C) or standards inconsistencies (D) can be found during reviews, they are secondary objectives. Application failures (A) are typically identified during dynamic testing, not during reviews.Impact analysis মানে পরিবর্তনের প্রভাব যাচাই—কতটুকু রিগ্রেশন দরকার তা নির্ধারণ।

22.Use case testing হলো একটি functional testing technique, যেখানে বাস্তব জীবনের ব্যবহারকারীর কাজের ধারা (process flow) অনুযায়ী টেস্ট করা হয়, এছাড়াও, যেহেতু use case testing-এ বিভিন্ন কম্পোনেন্ট একসাথে কাজ করে, তাই integration defects ধরা পড়ে. 

23. Test independence মানে হলো টেস্টিং কাজটি সেই ব্যক্তি করবেন না যিনি কোড লিখেছেন বা ডকুমেন্ট তৈরি করেছেন। এর মূল সুবিধা হলো author bias এড়ানো—যিনি কাজটি করেছেন তিনি নিজের ভুল সহজে ধরতে পারেন না, কিন্তু স্বাধীন টেস্টাররা নিরপেক্ষভাবে ভুল খুঁজে বের করতে পারেন.

24.Test planning: হলো পুরো টেস্ট প্রক্রিয়ার ভিত্তি। এতে অন্তর্ভুক্ত হয়:Test objectives নির্ধারণ → কী অর্জন করতে চাই তা স্পষ্ট করা। Test scope নির্ধারণ → কোন ফিচার বা মডিউল টেস্ট হবে, কোনগুলো বাদ যাবে। Test approach নির্বাচন → Black-box, White-box, Risk-based ইত্যাদি। Resource planning → টেস্টার, টুলস, environment, এবং অন্যান্য প্রয়োজনীয় রিসোর্স।Schedule তৈরি → টেস্টিং-এর টাইমলাইন, milestones, এবং deadlines।Risk identification এবং mitigation plan → সম্ভাব্য ঝুঁকি চিহ্নিত করে সমাধান পরিকল্পনা। Entry এবং Exit criteria নির্ধারণ → কখন টেস্ট শুরু হবে এবং কখন শেষ হবে। Deliverables নির্ধারণ → Test plan, test cases, defect reports, summary reports ইত্যাদি। Stakeholder communication plan → কাকে, কখন, কীভাবে রিপোর্ট করা হবে

25. Test Monitoring & Control → Progress tracking, defect metrics, coverage, resource usage, corrective actions, re-planning।

26. Test Analysis: Test basis বিশ্লেষণ করে testable features বের করা হয়। Test conditions define ও prioritize করা হয়, সাথে risks ও risk levels নির্ধারণ করা হয়। Defects ও testability assess করা হয়। প্রশ্নের উত্তর দেয়: “What to test?

27. Test Design: Test conditions elaboration → test cases ও testware তৈরি। Coverage items চিহ্নিত করা হয়। Test data requirements define করা হয়। Test environment design করা হয় এবং প্রয়োজনীয় infrastructure/tools নির্ধারণ করা হয়। প্রশ্নের উত্তর দেয়: “How to test?

28. Test Implementation: Testware তৈরি বা acquire করা হয় (যেমন test data)। Test cases → test procedures → test suites এ সংগঠিত করা হয়। Manual ও automated scripts তৈরি হয়। Test execution schedule তৈরি হয়। Test environment build ও verify করা হয়

29. Test Execution: Test runs execute করা হয় (manual বা automated)। Actual বনাম expected results তুলনা করা হয়। Results log করা হয়। Anomalies analyze করে defect cause বের করা হয়। Failures ভিত্তিক defect report করা হয়.

30. Test Completion/Closure Activities: Milestone (release, iteration end, test level completion)-এ সম্পন্ন হয়। Unresolved defects, change requests, backlog items handle করা হয়। Useful testware archive handover করা হয়। Test environment shutdown করা হয়। Lessons learned ও improvement identify করা হয়। Test completion report তৈরি ও stakeholders-কে communicate করা হয়।

31. Severity (গুরুত্ব/প্রভাব) একটি defect বা bug সফটওয়্যারের কার্যকারিতা কতটা ক্ষতিগ্রস্ত করছে। নির্ধারণ করে: সাধারণত QA বা টেস্টার।

32. Priority (অগ্রাধিকার/জরুরিতা) কোন defect আগে ঠিক করতে হবে, অর্থাৎ bug fixing-এর ক্রম। নির্ধারণ করে: সাধারণত Project Manager বা Client-এর সাথে আলোচনা করে।

33. Non-functional software quality characteristics:  Performance efficiency, Compatibility, Usability, Reliability, Security, Maintainability, Portability.

34. Test Level: Component testing (also known as unit testing), Component integration testing (also known as unit integration testing), System testing, System integration testing, Acceptance testing.

35. Deliverable base-lining (ডেলিভারেবল বেসলাইনিং): এটি মূলত Configuration Management বা Project Management-এর অংশ। কোনো ডেলিভারেবল (যেমন ডকুমেন্ট, কোড, ডিজাইন) নির্দিষ্ট সময়ে "ফাইনাল" বা "বেসলাইন" করে রাখা হয় যাতে পরবর্তীতে পরিবর্তন ট্র্যাক করা যায়। এটি Defect Management-এর কার্যক্রম নয়।

36.Stress Testing (স্ট্রেস টেস্টিং): যখন সিস্টেমকে তার স্বাভাবিক সীমার বাইরে ঠেলে দেওয়া হয়—অতিরিক্ত ব্যবহারকারী, অতিরিক্ত ডেটা, বা সীমাবদ্ধতা অতিক্রম করা হয়—তখন দেখা হয় সিস্টেম ভেঙে পড়ে কিনা বা কিভাবে প্রতিক্রিয়া দেয়। এটি মূলত সিস্টেমের স্থিতিশীলতা যাচাই করে। Load Testing (লোড টেস্টিং): নির্দিষ্ট সংখ্যক ব্যবহারকারী বা ট্রানজ্যাকশন দিয়ে সিস্টেমের পারফরম্যান্স যাচাই করা হয়। এটি সীমার ভেতরে থাকে। Performance Testing (পারফরম্যান্স টেস্টিং): সিস্টেমের গতি, প্রতিক্রিয়া সময়, এবং দক্ষতা যাচাই করা হয়। Volume Testing (ভলিউম টেস্টিং): প্রচুর পরিমাণে ডেটা দিয়ে সিস্টেম কিভাবে কাজ করে তা দেখা হয়।

37. Random Testing (র‌্যান্ডম টেস্টিং): এখানে টেস্টার কোনো পরিকল্পনা ছাড়া এলোমেলোভাবে ইনপুট দেয়। Gorilla Testing (গরিলা টেস্টিং): একটি নির্দিষ্ট মডিউলকে বারবার টেস্ট করা হয় অনেকটা জোর করে চাপ দেওয়ার মতো । Adhoc Testing (অ্যাডহক টেস্টিং): অভিজ্ঞতার ভিত্তিতে, কোনো ডকুমেন্টেড টেস্ট কেস ছাড়া টেস্ট করা হয়। Dumb Monkey Testing (ডাম্ব মাঙ্কি টেস্টিং): টেস্টার সফটওয়্যার সম্পর্কে কিছুই জানে না, শুধু এলোমেলোভাবে ক্লিক বা টাইপ করে। Structural Testing → নিশ্চিতভাবেই বাস্তব পরিস্থিতি mimic করে না।

38. Project Plan (প্রজেক্ট প্ল্যান): টেস্ট প্ল্যান তৈরি করার জন্য সবচেয়ে গুরুত্বপূর্ণ ইনপুট আসে এখান থেকে। কারণ প্রজেক্ট প্ল্যানে সময়সূচি, রিসোর্স, ডেলিভারেবল, এবং কার্যক্রমের বিবরণ থাকে। টেস্ট প্ল্যানকে সেই অনুযায়ী সাজাতে হয়। Business Plan (বিজনেস প্ল্যান): এটি মূলত ব্যবসায়িক লক্ষ্য ও কৌশল নির্ধারণ করে, কিন্তু সরাসরি টেস্ট প্ল্যানের ইনপুট হিসেবে ব্যবহৃত হয় না। Support Plan (সাপোর্ট প্ল্যান): এটি সাপোর্ট ও মেইনটেন্যান্স সম্পর্কিত, টেস্ট প্ল্যানের জন্য প্রধান ইনপুট নয়।

39.Code coverage (কোড কতটা টেস্টে কভার হয়েছে): এটি ইন্সপেকশন Inspection দ্বারা জানা যায় না। এটি জানতে হয় ডাইনামিক টেস্টিং টুল বা কোড কভারেজ অ্যানালাইসিস ব্যবহার করে। সাধারণত সফটওয়্যার ডেভেলপমেন্ট খরচের প্রায় অর্ধেক টেস্টিং-এ চলে যায়। 40-50% → এটি বাস্তবসম্মত সংখ্যা। 

40.Testability বলতে বোঝায় একটি সিস্টেমকে কতটা সহজে ও কার্যকরভাবে পরীক্ষা করা যায়। এর জন্য কিছু মূল বৈশিষ্ট্য থাকে: Operability (অপারেবিলিটি): সিস্টেম যদি ভালোভাবে কাজ করে, তাহলে টেস্ট করা সহজ হয়। Observability (অবজারভেবিলিটি): সিস্টেমের ভেতরের অবস্থা ও আউটপুট সহজে দেখা যায় কিনা, এটি টেস্টিং-এর জন্য জরুরি। Simplicity (সিমপ্লিসিটি): সিস্টেম যত সহজ হবে, টেস্ট করা তত সহজ হবে। কিন্তু— Robustness (রোবাস্টনেস): এটি মূলত সিস্টেমের স্থায়িত্ব ও ত্রুটি সহনশীলতার বৈশিষ্ট্য। এটি সিস্টেমের গুণমানের অংশ, কিন্তু সরাসরি Testability-এর বৈশিষ্ট্য নয়।

41. Able to write software (সফটওয়্যার লিখতে পারা): যদিও প্রোগ্রামিং জ্ঞান থাকলে টেস্ট অটোমেশন বা স্ক্রিপ্টিং-এ সুবিধা হয়, কিন্তু একজন ভালো টেস্টারের জন্য এটি সবচেয়ে কম প্রয়োজনীয় দক্ষতা। কারণ টেস্টিং মূলত বিশ্লেষণ, পর্যবেক্ষণ ও রিপোর্টিং-এর ওপর নির্ভর করে। Good Skills: Being diplomatic, Attention to detail, Able to be relied on. 
একটি নির্ভরযোগ্য (reliable) সিস্টেম হবে এমন একটি সিস্টেম যা- Is unlikely to cause a failure (ত্রুটি ঘটানোর সম্ভাবনা কম): নির্ভরযোগ্য সিস্টেমের মূল বৈশিষ্ট্য হলো এটি ব্যর্থতা ঘটানোর সম্ভাবনা কম থাকে।

42.“A form of control flow testing based on decision outcomes.” এটি Decision Testing, A test technique which combines combinations of inputs…” এটি Decision Table Testing, এটি State Transition Testing কারণ system এর behaviour current state বা previous state এর উপর depend করে, “Testing carried out … to achieve specific test objectives, possibly to complement structured testing.” এটি বোঝায় Exploratory Testing.

43. Test Manager এর দায়িত্ব হলো টেস্ট প্ল্যান তৈরি করা, রিসোর্স ম্যানেজ করা, ডিফেক্ট রিপোর্ট করা, রিস্ক অ্যানালাইসিসে তথ্য দেওয়া ইত্যাদি। কিন্তু সিস্টেম রিলিজ সাইন-অফ করা তাদের কাজ নয়। এটা সাধারণত প্রজেক্ট ম্যানেজার বা কাস্টমার/স্টেকহোল্ডার এর দায়িত্ব।

44. Unreachable code সাধারণত Static Testing এ পাওয়া যায় এবং সবচেয়ে কার্যকরভাবে পাওয়া যায় Static Analysis Tool দিয়ে and Code reviews → মানুষ ম্যানুয়ালি দেখে, কিন্তু সব unreachable code ধরা কঠিন। LCSAJ-Linear Code Sequence and Jump এটি code structure analyse করে testing করা হয়। অর্থাৎ এটি white box technique.

45. Configuration Control-Controlled library (যাকে সাধারণত Configuration Management System বা CM library বলা হয়) এর মধ্যে সব configuration items রাখা হয়। এর provision ও management মানে হলো কে access করবে, কীভাবে update হবে, এবং কীভাবে version control হবে—এসব নিয়ন্ত্রণ করা।

46. Test Condition: একটি নির্দিষ্ট aspect বা feature যা টেস্ট করা দরকার। এগুলো derive করা হয় specifications, requirements, design documents থেকে। অর্থাৎ, টেস্ট কন্ডিশন সবসময় source documents (specifications) থেকে আসে। Driver er synonym: Test harness, Scaffolding → ✅ সঠিক, . 

47. Incremental testing approaches মানে হলো মডিউলগুলো ধাপে ধাপে integrate করে টেস্ট করা। এর মধ্যে থাকে: Top-down → উপরের মডিউল থেকে শুরু করে নিচের দিকে ধাপে ধাপে integrate করা। Bottom-up → নিচের মডিউল থেকে শুরু করে উপরের দিকে ধাপে ধাপে integrate করা। Functional incrementation → ফাংশনালিটি অনুযায়ী ধাপে ধাপে integrate করা। Big-bang → সব মডিউল একসাথে integrate করে টেস্ট করা হয়। এটা incremental নয়, বরং non-incremental approach।

48. IEEE 829 (Standard for Software Test Documentation) এ incident report এর অফিসিয়াল নাম হলো Test Incident Report। প্রশ্নে IEEE 1044 বা anomaly classification বলা থাকে → Anomaly Report হবে। Testing এর মূল উদ্দেশ্য হলো defect খুঁজে বের করা, এবং সফটওয়্যার প্রত্যাশিতভাবে কাজ করছে কিনা তা যাচাই করা। Testing কখনোই প্রমাণ করতে পারে না যে সফটওয়্যার defect-free, বরং defect খুঁজে বের করার চেষ্টা করে.

49. High-level design → Integration tests, Modules এর interaction design হয়, তাই integration testing এর সাথে মিলে যায়। Low-level design → System tests, Detailed design পুরো সিস্টেমের behavior এর সাথে সম্পর্কিত, তাই system testing। Requirement → Acceptance, HLD → Integration (module interaction), LLD → System (whole system behavior), Code → Unit (individual functions).

50. Requirements capture বা analysis করার জন্য capture & replay tool ব্যবহার করা হয় না। Dynamic Analysis Tool: এটি প্রোগ্রাম চলাকালীন সময়ে (runtime) মেমরি ব্যবহার, পয়েন্টার অ্যাক্সেস, এবং সম্ভাব্য মেমরি লিক শনাক্ত করে। তাই মেমরি লিক ও unassigned pointers খুঁজে বের করার জন্য এটি ব্যবহৃত হয়। Viability (টিকে থাকার ক্ষমতা): এটি মূলত ব্যবসায়িক বা অর্থনৈতিক দিক নির্দেশ করে, সফটওয়্যার কোয়ালিটির বৈশিষ্ট্য নয়।

51. Test estimation → Effort required to perform activities টেস্টিং-এর জন্য কতটা সময়, রিসোর্স ও প্রচেষ্টা লাগবে তা অনুমান করা হয়। Test control → Reallocation of resources টেস্টিং চলাকালীন প্রয়োজন অনুযায়ী রিসোর্স পুনর্বণ্টন বা পরিবর্তন করা হয়। Test monitoring → Measures of tracking process টেস্টিং প্রক্রিয়ার অগ্রগতি ও কার্যকারিতা ট্র্যাক করা হয়।

52. Configuration identification → Requires that all CI’s and their versions in the system are known কোন কোন Configuration Item (CI) আছে এবং তাদের version কী, তা সনাক্ত করা হয়। Configuration control → Maintains of CI’s in a library CI গুলোকে লাইব্রেরিতে সংরক্ষণ ও নিয়ন্ত্রণ করা হয়। Status reporting → c. Function recording and tracking problems CI-এর অবস্থা, পরিবর্তন ও সমস্যা ট্র্যাক করা হয়। Configuration auditing → Checks on the contents of the library লাইব্রেরির কনটেন্ট যাচাই করা হয়, সঠিকভাবে সংরক্ষিত কিনা তা নিশ্চিত করা হয়।

53. Test execution-> এখানে CAST টুল সবচেয়ে কার্যকর। কারণ টেস্ট কেসগুলো অটোমেটেডভাবে চালানো যায়, বারবার রান করা যায়, এবং ম্যানুয়াল প্রচেষ্টা অনেক কমে যায়। ফলে সময় ও খরচ উভয়ই ব্যাপকভাবে সাশ্রয় হয়। Error Guessing: এটি টেস্টারের অভিজ্ঞতা ও intuition-এর উপর ভিত্তি করে করা হয়। টেস্টার অনুমান করে কোথায় ত্রুটি থাকতে পারে এবং অতিরিক্ত টেস্ট করে। এটি rigorous testing-এর supplement হিসেবে ব্যবহৃত হয়।

54. Test design techniques (যেমন equivalence partitioning, boundary value analysis, decision table testing ইত্যাদি) ব্যবহার করা হয় টেস্ট কেস বাছাই করার জন্য। 
IEEE 829: এটি Standard for Software Test Documentation — টেস্ট প্ল্যান, টেস্ট কেস, টেস্ট রিপোর্ট ইত্যাদি ডকুমেন্টেশনের জন্য ব্যবহৃত হয়। IEEE 610: এটি Standard Glossary of Software Engineering Terminology — সাধারণ সফটওয়্যার ইঞ্জিনিয়ারিং টার্মসের জন্য, কিন্তু সরাসরি টেস্টিং স্ট্যান্ডার্ড নয়। BS 7925-1: এটি Software Testing Vocabulary — টেস্টিং টার্মিনোলজি নির্ধারণ করে। BS 7925-2: এটি Software Component Testing Standard — টেস্টিং টেকনিক ও প্র্যাকটিস নির্ধারণ .

55. Requirements faults : যদি রিকোয়ারমেন্টে ভুল থাকে এবং তা পরে ধরা পড়ে, তখন সেই ভুল ইতিমধ্যেই ডিজাইন, কোড, টেস্ট কেস, ডকুমেন্টেশন সবকিছুতে ছড়িয়ে পড়ে। ফলে এটি ঠিক করতে সবচেয়ে বেশি খরচ হয়। Coverage measurement —  is a partial measure of test thoroughness. Code inspection: এটি static testing-এর একটি উদাহরণ। কোড execute না করে রিভিউ করা হয়। Interface bug = যখন দুটি কম্পোনেন্টের মধ্যে data mismatch বা communication problem হয়। অনেক কম্পাইলার static analysis-এর কিছু ফিচার দেয় (যেমন syntax checking, type checking, warning generation)।

56. Branch/Decision coverage ≠ Condition coverage, 100% branch/decision coverage becomes to 100% statement coverage,

57. V-Model is a software development model that illustrates how testing activities integrate with software development phases.incremental integration has better early defects screening and isolation ability compare to big-bang integration. Maintenance testing - should include operational tests of the new environment as well as of the changed software.

58. Big bang testing সব কম্পোনেন্ট একসাথে combine করে টেস্ট করা হয়। কোনো incremental step থাকে না, অর্থাৎ ধাপে ধাপে integration করা হয় না। এর ফলে defect isolation কঠিন হয়, কারণ একসাথে অনেক module integrate হয়। Moderator → Review মিটিং পরিচালনা করে, কিন্তু তার জন্য technical/business background থাকা আবশ্যক নয়। Author → যিনি ডকুমেন্ট বা কোড লিখেছেন। তিনি অবশ্যই technical background রাখেন, কিন্তু business background থাকা বাধ্যতামূলক নয়। Reviewer → যিনি review করেন। Reviewer এর technical এবং business দুই ধরনের background থাকা দরকার, যাতে তিনি কোড/ডকুমেন্টের correctness এবং business requirements এর সাথে alignment যাচাই করতে পারেন। 

59.Penetration Testing → Security vulnerabilities খুঁজে বের করার জন্য করা হয়. Syntax Testing-Specification languages (যেমন grammar, BNF notation, বা formal syntax rules) এর উপর ভিত্তি করে test cases তৈরি করা হয়। State Transition Testing → State machine এর উপর ভিত্তি করে test cases তৈরি হয়. White Box Testing কে বিভিন্ন নামে ডাকা হয়: Glass box testing, Clear box testing কখনো কখনো Structural testing নামেও উল্লেখ করা হয়। মূল ধারণা হলো: সফটওয়্যারের internal structure, logic, এবং code paths দেখে টেস্ট করা।

60. Design stages এ verification activities সাধারণত হয়: Reviewing → Design documents, specifications, architecture review করা। Inspecting → Formal inspection process, যেখানে defects খুঁজে বের করা হয়। QC (Quality Control) হলো একটি end-of-phase activity, যেখানে প্রতিটি development phase শেষ হওয়ার পর deliverables যাচাই করা হয়। QC এর মাধ্যমে নিশ্চিত করা হয় যে product বা deliverable নির্ধারিত quality standards পূরণ করছে। Preparing and automating test cases before coding" আসলে Test First Approach এবং Test-driven development — দুটোই বোঝায়।

61.  কম configuration এবং কম রাউন্ডে manual testing সহজ ও কার্যকর। সব incident fix করা হয় না। কিছু incident হতে পারে: Minor cosmetic issue (যা business-critical নয়),Known limitation বা low-priority defect,এমন কিছু যা fixing করতে গেলে বেশি খরচ/সময় লাগবে কিন্তু business impact কম। defect/fault early detection = cheap, Late detection= high cost. Capture-Replay Tool = Record user actions → Replay automatically. V-model -> Design of verification কে অন্তর্ভুক্ত করে।

62. Oracle assumption মানে হলো: টেস্টার ধরে নেয় যে সে প্রতিটি টেস্টের সঠিক outcome (expected result) নির্ধারণ করতে পারবে। অর্থাৎ, যখন টেস্ট চালানো হয়, তখন টেস্টার expected result জানে এবং actual result এর সাথে তুলনা করতে পারে। Code inspections হলো static testing technique, যেখানে কোড execute না করেই ত্রুটি খুঁজে বের করা যায়।

63. Test case design করার সময় কার্যক্রমের সঠিক ক্রম হলো: Analyse requirements and specifications to determine test conditions-প্রথমে requirement ও specification দেখে test condition বের করতে হয়। Elaborate and describe test cases in detail by using test design techniques-এরপর test design techniques ব্যবহার করে বিস্তারিত test case তৈরি করা হয়।
Specify expected results-প্রতিটি test case-এর জন্য expected result নির্ধারণ করতে হয়। Specify the order of test case execution-সবশেষে execution order ঠিক করা হয়।

64. Test procedure specification → এখানে test case execution-এর sequence/order নির্ধারণ করা হয়। অর্থাৎ কোন test আগে চলবে, কোনটা পরে চলবে। Test design specification → Test conditions, coverage items, এবং high-level design নির্ধারণ করে। Test case specification → প্রতিটি test case-এর input, precondition, expected result ইত্যাদি বর্ণনা করে। Test plan → Testing-এর overall objectives, scope, resources, schedule ইত্যাদি নির্ধারণ করে। Reliability সবসময় বাস্তব ব্যবহার frequency অনুযায়ী test case নির্বাচন করে।
